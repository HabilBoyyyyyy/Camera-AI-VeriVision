1. Use case overview
This project is based on WINTEQ's "Camera AI – Dynamic Software" use case: a flexible, on-premise AI vision platform that lets industries deploy, train, and scale visual inspection applications without engineers rewriting code for every new customer.
1.1 Business problem
The current Camera AI application relies heavily on hardcoded configurations and business logic, requiring direct source code changes by the development team.
This makes the solution difficult to adapt and replicate across different customers, limiting scalability.
Maintaining and deploying customized versions increases both development effort and long-term support costs.
1.2 Goal
Transform Camera AI into a flexible, configurable platform that adapts to different customer needs.
Enable users to configure workflows and integrate with external systems without code changes.
Reduce dependency on the development team for routine customization requests.
1.3 Original key features (as briefed)
Modular workflow architecture for flexible process design.
Configuration-driven setup to minimize or eliminate hardcoded logic.
Template management for different inspection and business use cases.
Integration framework for seamless connectivity with external systems.
User-friendly parameter management through the application interface.
1.4 What each feature actually means in implementation terms
These five bullet points are business language. Translated into concrete engineering terms:
Modular workflow architecture: the inspection process is broken into small, independent steps (resize, crop, classify, decide, log) that can be reordered or swapped without touching other steps. In code, this usually means a list of step objects processed by one generic loop, rather than one long function that does everything in a fixed order.
Configuration-driven setup: the behavior of the system is controlled by data (JSON/YAML) that is read at runtime, not by source code that has to be recompiled or redeployed. This is the single most important technical concept in this project — see Section 9 for the full schema.
Template management: templates are just saved configurations (rows in a database). Creating a new inspection type is a database insert, not a code change.
Integration framework: a defined, predictable way for the system to talk to the outside world (e.g. a REST API endpoint or webhook), so external systems don't need custom code written for them either.
User-friendly parameter management: forms and UI controls that let a non-programmer edit the JSON configuration indirectly, without ever seeing raw JSON.

2.2 Reasoning behind each cut
Cutting scope isn't just "doing less" — each cut below is a deliberate trade-off, so you can explain it confidently if an evaluator asks "why didn't you build X?"
Real industrial protocols (Profinet, OPC UA, Modbus, GigE Vision): these require physical hardware and vendor-specific SDKs that a student team won't have access to. A webcam is functionally equivalent for demonstrating the AI + configuration logic, which is what's actually being evaluated.
Multiple AI task types: each additional task type (detection, OCR, segmentation) is effectively a separate machine learning project with its own data labeling, training, and evaluation cycle. Doing one well beats doing four poorly.
Drag-and-drop workflow builder: this is a significant frontend engineering effort on its own (drag events, visual connectors, validation) that doesn't change the underlying architecture — a form-based UI proves the same configuration-driven concept with a fraction of the build time.
Multi-camera support: mostly adds UI complexity (managing multiple video streams and layouts) without teaching anything new about the core concept.
Authentication / multi-role access: a real auth system (password hashing, sessions/tokens, permission checks) is a substantial security-sensitive feature; getting it wrong is worse than not having it for a prototype.
Report export and real-time alerts: these are polish features that extend an already-working pipeline. They're good "if time remains" additions but never should block the core demo.

3. Web accessibility: desktop and mobile
The system has two distinct layers, and only one of them can live in the browser.
3.1 Layer A — camera / sensor communication (hardware layer)
Protocols like GigE Vision, Profinet, EtherNet/IP, Modbus TCP, OPC UA, MQTT, and CameraLink cannot be accessed directly from a web browser (browsers are sandboxed for security). This layer must run as a backend service/agent installed on-premise, which connects to the camera(s), runs AI inference, and sends results to the web app via a normal API (REST/WebSocket).
Why browsers can't do this directly: browsers deliberately restrict access to raw network sockets, serial ports, and industrial fieldbus protocols as a security measure — a malicious website should never be able to reach into a factory network. This isn't a limitation you can code around; it's intentional sandboxing built into every modern browser. That's why an on-premise backend agent (a normal program running outside the browser) is required for this layer.
Phase 2 addition: a PLC fits into this same hardware layer, on both ends of the pipeline — as a trigger (a PLC signal starts an inspection) and as an output actuator (the backend agent writes the OK/NG result back to the PLC over Modbus TCP). Neither can happen inside the browser, for the same sandboxing reason above; both would live in the on-premise backend agent alongside the camera drivers.
3.2 Layer B — configuration, monitoring, dashboard (software layer)
This layer can be fully web-based. A responsive web app (CSS breakpoints, flexible grids) adapts to desktop, tablet, and mobile. Optionally, building it as a Progressive Web App (PWA) lets users "install" it on their phone home screen with some offline capability, without needing separate iOS/Android apps.
Practical responsive design technique: use CSS media queries to change the layout at defined breakpoints, e.g. a single column on phones and a multi-column grid on desktop. The frontend scaffold's stylesheet uses exactly this pattern: a mobile-first single-column grid that switches to two columns above 768px screen width — small enough to write, easy enough to explain in a presentation, and it satisfies the "works on desktop and mobile" requirement without extra libraries.
Summary: the camera + AI model does its work on a machine physically near the camera (the "edge"), and the web app is the control panel/dashboard that any device with a browser can open to configure, monitor, and view results. This is a standard industrial pattern.
4. Simple explanation

Imagine a factory that uses cameras to automatically check if a product has defects (scratches, missing parts, wrong assembly, etc). 5. Recommended technical solution
Frontend: Next.js (per final team decision — Nabil's role) for the configuration/dashboard UI.
Backend/API: Python with FastAPI — strong AI/ML library support, needed since the inference engine uses PyTorch/TensorFlow-based models.
AI inference engine: a pre-trained or fine-tuned classification model (e.g. MobileNet/ResNet via transfer learning) for OK/NG sorting.
Database: PostgreSQL or SQLite — stores inspection templates, results, and logs.
Workflow configuration: each inspection "workflow" is stored as a JSON definition rather than code — this is the core mechanism that satisfies "no hardcoded logic" (see Section 9).
Camera simulation: a webcam or pre-recorded image/video dataset stands in for a real industrial camera — an acceptable substitution for an academic project.
Phase 2 hardware integration (target, not built for the bootcamp demo): pymodbus for PLC communication over Modbus TCP, plus an industrial camera SDK (e.g. Harvesters/Aravis for GigE Ethernet cameras, or a vendor USB3 Vision SDK) running inside the same on-premise backend agent as Layer A.
5.1 Why these choices, and what the alternatives were
Chosen
Alternatives considered, and why not
FastAPI (backend)
Django: heavier, includes a lot of built-in features (admin panel, ORM) you don't need for an API-only backend. Flask: lighter but requires more manual setup for data validation, which FastAPI gives you for free via Pydantic.
Next.js (frontend)
Plain React (Create React App / Vite): works fine too, but Next.js gives built-in routing and simpler deployment, which matters more once the team is short on time.
SQLite for development, PostgreSQL optional for production
MongoDB (NoSQL): works, but since templates and results both have a clear tabular structure (columns, relationships), a relational database is a more natural fit and easier to reason about for a beginner.
Transfer learning on MobileNet/ResNet
Training a model completely from scratch: requires far more labeled data and compute time than a 3-week project allows. Transfer learning re-uses a model already trained on millions of general images and only retrains the final layers on your specific defect dataset.
Beginner note on transfer learning: think of it like hiring someone who already knows how to see shapes, edges, and textures in general, and just teaching them the specific difference between "scratched" and "not scratched" — instead of teaching someone to see from zero. This is why it needs far less data and time than training from scratch.

9. Configurable workflow JSON schema
This is the core mechanism that makes the system "configurable instead of hardcoded." Instead of writing per-customer if/else logic, inspection logic is stored as data (JSON), and the backend reads that data and executes steps dynamically.
{
  "template_id": "tmpl_0001",
  "template_name": "Bracket Scratch Inspection",
  "description": "Detects surface scratches on metal brackets",
  "input_source": {
    "type": "webcam",
    "device_id": 0
  },
  "preprocessing_steps": [
    { "step_type": "resize", "params": { "width": 224, "height": 224 } },
    { "step_type": "crop", "params": { "x": 50, "y": 50, "width": 400, "height": 400 } }
  ],
  "inspection_step": {
    "step_type": "classification",
    "model_id": "model_scratch_v1",
    "confidence_threshold": 0.80
  },
  "decision_rule": {
    "pass_label": "OK",
    "fail_label": "NG",
    "min_confidence_to_pass": 0.80
  },
  "output_actions": [
    { "action_type": "save_result", "params": { "store_image": true } },
    { "action_type": "log_event", "params": { "log_level": "info" } }
  ]
}
Why this structure works: preprocessing_steps and output_actions are arrays, so the backend can loop through them generically — adding a new step type later means adding a new case to one loop, not writing a new function per customer. decision_rule separates the AI's raw confidence output from the business pass/fail threshold, so different customers can use different thresholds without retraining the model.
9.1 The generic executor, in actual Python
This is what makes the schema above more than just documentation — it's the real function that reads a template and runs it. Notice there is no if/else chain per customer; the same function handles every template.
def run_workflow(template: dict, image):
    """Reads a template dict (parsed from JSON) and executes each step in order."""
 
    # 1. Run preprocessing steps in the order they appear in the template
    for step in template["preprocessing_steps"]:
        step_type = step["step_type"]
        params = step["params"]
 
        if step_type == "resize":
            image = resize_image(image, params["width"], params["height"])
        elif step_type == "crop":
            image = crop_image(image, params)
        # Adding a new preprocessing step later just means adding
        # another elif here - nothing else in the codebase changes.
 
    # 2. Run the AI inference step
    inspection = template["inspection_step"]
    label, confidence = classify_image(image, model_id=inspection["model_id"])
 
    # 3. Apply the business decision rule from the template
    rule = template["decision_rule"]
    passed = confidence >= rule["min_confidence_to_pass"]
    result = rule["pass_label"] if passed else rule["fail_label"]
 
    # 4. Run output actions (save to DB, log, etc.)
    for action in template["output_actions"]:
        run_action(action["action_type"], action["params"], result)
 
    return {"label": label, "confidence": confidence, "result": result}
This function is what lives in backend/app/services/workflow_executor.py in the project scaffold. Once it's written and tested once, creating a brand-new inspection type for a new customer never touches this file again — it's purely a matter of saving a new template JSON to the database.

The Phase 1 schema in Section 9 covers webcam input and manual model selection. The client’s full vision extends the same JSON shape — nothing about the executor in Section 9.1 needs to change, only new fields are read by new step handlers:

  },
  "inspection_step": {
    "step_type": "classification",
    "model_id": "model_housing_crack_v1",

    "fail_label": "NG",

    { "action_type": "save_result", "params": { "store_image": true } },
    { "action_type": "log_event", "params": { "log_level": "info" } },

Two additions do the real work here. First, "trigger" decouples what starts an inspection (a click, or a PLC register flipping) from the executor itself — the executor in Section 9.1 does not need to know which one fired it. Second, "dataset_source.auto_train" is what turns "upload a new dataset" into a working model without a human touching code: it points the training pipeline (Section 8) at a dataset instead of requiring the AI teammate to run a notebook by hand.
None of this is built for the July 23 demo — it is documented here so the Phase 1 architecture can grow into it without a rewrite.
10. System architecture
Client devices and a camera feed into a backend server containing an API layer, an AI inference engine, and a database.
Client devices (desktop or mobile browser) only ever talk to the API layer.
The camera feeds images into the API layer, which forwards them to the AI inference engine.
Both the API layer and the inference engine write to the database — templates on creation, results on every inspection run.
Client devices and the camera can be swapped or extended later without touching backend logic — this is what makes the "configuration-driven" claim true in the architecture, not just in the pitch.
10.1 Step-by-step walkthrough of one inspection
Tracing a single inspection from start to finish through the diagram above:
1. The operator opens the dashboard in a browser (desktop or mobile) and clicks "Run inspection."
2. The browser captures a frame from the camera feed (see Section 12) and sends it to the API layer as an HTTP request.
3. The API layer looks up the selected template from the database — this is the JSON configuration from Section 9.
4. The API layer calls the workflow executor, which reads the template and runs each preprocessing step on the image.
5. The executor calls the AI inference engine, passing the processed image and the model_id from the template.
6. The AI inference engine returns a label and confidence score back to the executor.
7. The executor applies the decision_rule from the template to turn that confidence score into a pass/fail result.
8. The result is saved to the database and returned back through the API layer to the browser.
9. The dashboard updates the results table with the new entry, without the page needing to reload.
Every one of these nine steps is the same regardless of which template is used — only the data inside the template changes. That consistency is exactly what "configuration-driven, not hardcoded" looks like when it's actually running.

12. Camera integration: current state & gaps
The live camera preview (getUserMedia) is working, but a genuine gap exists between "camera shows up on screen" and "camera image gets inspected": there was no step that captures a still frame and sends it to the backend for inference.
Frontend gap: a function to grab a frame from the <video> element (via <canvas>), convert it to an image, and send it via fetch() to the backend.
Backend gap: the inspection run endpoint needs to actually accept that image as a file upload (FastAPI UploadFile) instead of ignoring it.
Both gaps have been added as explicit tasks in the Jira plan (Section 13, highlighted rows).
12.1 Closing the frontend gap: capturing a frame
The <video> element only shows a live stream — it can't be sent directly over the network. The trick is to draw the current video frame onto a hidden <canvas> element, which can export its content as an image file. This function would be added to dashboard.js:
function captureFrame() {
  const video = document.getElementById("camera-feed");
  const canvas = document.createElement("canvas");
  canvas.width = video.videoWidth;
  canvas.height = video.videoHeight;
 
  // Draw the current video frame onto the canvas
  const ctx = canvas.getContext("2d");
  ctx.drawImage(video, 0, 0, canvas.width, canvas.height);
 
  // Convert the canvas content into an image file (Blob)
  return new Promise((resolve) => {
    canvas.toBlob((blob) => resolve(blob), "image/jpeg", 0.9);
  });
}
 
async function runInspectionWithImage(templateId) {
  const imageBlob = await captureFrame();
  const formData = new FormData();
  formData.append("image", imageBlob, "capture.jpg");
 
  const res = await fetch(
    `${API_BASE}/inspections/run/${templateId}`,
    { method: "POST", body: formData }
  );
  return res.json();
}
12.2 Closing the backend gap: accepting the image
FastAPI's UploadFile type handles file uploads with very little extra code. The inspection router needs updating so it actually receives and uses the image instead of ignoring it:
from fastapi import APIRouter, UploadFile, File
from PIL import Image
import io
 
@router.post("/run/{template_id}")
async def run(template_id: str, image: UploadFile = File(...)):
    # Read the uploaded bytes and turn them into an image object
    image_bytes = await image.read()
    pil_image = Image.open(io.BytesIO(image_bytes))
 
    # Pass the real captured image into the workflow executor
    # instead of the placeholder `None` used in the stub version
    result = run_inspection(template_id, pil_image)
    _results_log.append(result)
    return result
With these two pieces in place, the full loop described in Section 10.1 becomes real: a click on "Run inspection" genuinely captures the live camera frame, sends it to the backend, and gets back a result computed from that exact image.
