## What This Is

This schema defines how inspection workflows are stored as **data (JSON)** instead of code. It's the core mechanism that makes the system "configurable instead of hardcoded."

> [!IMPORTANT]
> **The golden rule**: Creating a new inspection type = saving a new JSON template to the database. The `workflow_executor.py` reads this JSON and runs it. Nobody touches source code.

## Schema Structure Overview

```
Template
├── template_id          "tmpl_0001"
├── template_name        "Bracket Scratch Inspection"
├── description          "Detects surface scratches..."
│
├── input_source ─────── WHERE the image comes from
│   ├── type             "webcam" | "image_upload"
│   └── device_id        0 (camera index)
│
├── preprocessing_steps ─ HOW to prepare the image (array, looped generically)
│   ├── [0] resize       { width: 224, height: 224 }
│   ├── [1] crop         { x, y, width, height }
│   └── [2] normalize    { mean: [...], std: [...] }
│
├── inspection_step ──── WHAT AI model to run
│   ├── step_type        "classification"
│   ├── model_id         "model_scratch_v1"
│   └── confidence_threshold  0.80
│
├── decision_rule ────── HOW to interpret the AI output
│   ├── pass_label       "OK"
│   ├── fail_label       "NG"
│   └── min_confidence_to_pass  0.80
│
└── output_actions ───── WHAT to do after the decision (array, looped generically)
    ├── [0] save_result  { store_image: true }
    ├── [1] log_event    { log_level: "info" }
    └── [2] trigger_alert { on: "fail" }
```

## Field-by-Field Reference

### `template_id` — string
- Format: `tmpl_XXXX` (auto-generated by backend)
- Example: `"tmpl_0001"`

### `template_name` — string
- Human-readable name displayed in the dashboard
- 3–100 characters
- Example: `"Bracket Scratch Inspection"`

### `description` — string
- Brief explanation of what this inspection checks
- Max 500 characters

### `input_source` — object

| Field | Type | Values | Description |
|-------|------|--------|-------------|
| `type` | string | `"webcam"` \| `"image_upload"` | Where the image comes from |
| `device_id` | int \| null | `0`, `1`, `2`... | Camera index. Only used with webcam. |

### `preprocessing_steps` — array of step objects

Each step has a `step_type` and `params`. The backend **loops through these in order** — adding a new step type later = adding one `elif` case, nothing else.

| step_type | params | Description |
|-----------|--------|-------------|
| `resize` | `{ width, height }` | Resize image to target dimensions (always 224×224 for our models) |
| `crop` | `{ x, y, width, height }` | Crop a region of interest before resizing |
| `normalize` | `{ mean: [R,G,B], std: [R,G,B] }` | Normalize pixel values. **Must use ImageNet stats** for transfer learning: `mean=[0.485, 0.456, 0.406]`, `std=[0.229, 0.224, 0.225]` |

> [!TIP]
> **Order matters!** Crop before resize if you want to focus on a specific area. Always normalize last.

### `inspection_step` — object

| Field | Type | Values | Description |
|-------|------|--------|-------------|
| `step_type` | string | `"classification"` | AI task type. Only classification for MVP. |
| `model_id` | string | e.g. `"model_scratch_v1"` | References a `.pt` file in `ai_models/` |
| `confidence_threshold` | float | `0.0` – `1.0` | Minimum confidence to consider prediction valid |

### `decision_rule` — object

This is what separates the AI's raw output from the **business decision**. Different customers can use different thresholds without retraining the model.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `pass_label` | string | `"OK"` | Label when part passes |
| `fail_label` | string | `"NG"` | Label when part fails |
| `min_confidence_to_pass` | float | — | If confidence ≥ this → OK, otherwise → NG |

**Example**: Model returns 78% confidence. Template A has threshold 0.75 → **OK**. Template B has threshold 0.80 → **NG**. Same model, different business rules.

### `output_actions` — array of action objects

| action_type | params | Description |
|-------------|--------|-------------|
| `save_result` | `{ store_image: true/false }` | Save result to database, optionally with the image |
| `log_event` | `{ log_level: "info"/"warning"/"error" }` | Write a log entry |
| `trigger_alert` | `{ on: "fail"/"always" }` | Trigger an alert notification |

## How the Backend Uses This

The workflow executor (from Section 9.1 of the project doc) reads a template and runs it generically:

```python
def run_workflow(template: dict, image):
    # 1. Preprocessing — loop through steps array
    for step in template["preprocessing_steps"]:
        if step["step_type"] == "resize":
            image = resize_image(image, step["params"]["width"], step["params"]["height"])
        elif step["step_type"] == "crop":
            image = crop_image(image, step["params"])
        elif step["step_type"] == "normalize":
            image = normalize_image(image, step["params"]["mean"], step["params"]["std"])

    # 2. AI inference
    label, confidence = classify_image(image, template["inspection_step"]["model_id"])

    # 3. Business decision
    rule = template["decision_rule"]
    passed = confidence >= rule["min_confidence_to_pass"]
    verdict = rule["pass_label"] if passed else rule["fail_label"]

    # 4. Output actions — loop through actions array
    for action in template["output_actions"]:
        run_action(action["action_type"], action["params"], verdict)

    return {"verdict": verdict, "confidence": confidence}
```

> [!NOTE]
> **No per-customer if/else logic.** The same function handles every template. New inspection type = new JSON row in the database.

## Extensibility (Future Work)

The schema is designed to grow without breaking existing templates:

| Extension | What changes | What stays the same |
|-----------|-------------|-------------------|
| New preprocessing step (e.g. `grayscale`) | Add one `elif` case in executor | All existing templates unchanged |
| New AI task type (e.g. `detection`) | Add to `inspection_step.step_type` enum | Existing `classification` templates still work |
| New output action (e.g. `send_email`) | Add one `elif` case in action runner | Existing actions unchanged |
| New customer | New template JSON in database | **Zero code changes** ✅ |

## For the Presentation

Key talking points this schema proves:

1. **"No hardcoded logic"** — the executor loops through arrays, doesn't have per-customer branches
2. **"Configuration-driven"** — business rules (thresholds, labels) live in JSON, not code
3. **"Extensible"** — adding a new preprocessing step = one elif, not a rewrite
4. **"Separation of concerns"** — AI output (confidence) is separate from business decision (threshold)
